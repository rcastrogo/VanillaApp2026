Aquí tienes cómo construir un compilador sencillo pero increíblemente potente usando Expresiones Regulares (Regex) y el constructor new Function().


type State = Record<string, any>;
type RenderFunction = (state: State) => string;

export function compileTemplate(template: string): RenderFunction {
  let code = "let out = '';\n";
  let cursor = 0;

  // Regex para atrapar: @if(...) { | @for(...) { | {variable} | }
  const regex = /@if\s*\(([^)]+)\)\s*\{|@for\s*\(([^)]+)\)\s*\{|\{([^}]+)\}|\}/g;
  let match;

  while ((match = regex.exec(template)) !== null) {
    // 1. Guardar el HTML estático puro (lo que hay antes de la coincidencia)
    const text = template.slice(cursor, match.index);
    if (text) {
      // Escapamos comillas invertidas para no romper el string
      code += `out += \`${text.replace(/`/g, '\\`')}\`;\n`;
    }

    // 2. Evaluar qué tipo de bloque encontramos y traducirlo a JS
    if (match[1]) { 
      // Es un @if
      code += `if (${match[1]}) {\n`;
    } else if (match[2]) { 
      // Es un @for (ej. match[2] será "item of items")
      code += `for (let ${match[2]}) {\n`;
    } else if (match[3]) { 
      // Es una interpolación {user.name}
      code += `out += ${match[3]};\n`;
    } else if (match[0] === '}') { 
      // Es el cierre de un bloque @if o @for
      code += `}\n`;
    }

    // Avanzamos el cursor
    cursor = match.index + match[0].length;
  }

  // 3. Añadir el HTML restante al final del template
  const rest = template.slice(cursor);
  if (rest) {
     code += `out += \`${rest.replace(/`/g, '\\`')}\`;\n`;
  }

  code += "return out;";

  // 4. La magia: Crear la función de renderizado.
  // Usamos destructuring dinámico para que las propiedades del estado
  // existan como variables locales dentro de la función evaluada.
  return function(state: State): string {
    const keys = Object.keys(state);
    const values = Object.values(state);
    
    // new Function(arg1, arg2, ..., cuerpoDeLaFuncion)
    const dynamicRender = new Function(...keys, code);
    
    return dynamicRender(...values);
  };
}

Ese es un problema clásico de "deuda técnica" al unificar sistemas. Tienes dos definiciones de lo mismo en sitios diferentes y, aunque son conceptualmente compatibles, sus tipos varían ligeramente (uno usa BaseComponent y el otro Component).

Para que tu sistema sea sólido (y para que "un anillo los gobierne a todos"), debemos consolidarlos. Aquí tienes cómo hacerlo:

1. Centraliza el modelo en component.model.ts
Este fichero debe ser la única fuente de verdad para lo que es un componente y cómo se crea.

TypeScript
// ../components/component.model.ts

export interface Component {
  [key: string]: any; 
  init?(): void;
  render(): HTMLElement;
  mounted?(): void;
  destroy?(): void;
  // Añadimos estas propiedades que usabas en la hidratación
  element?: HTMLElement;
  addCleanup?(fn: () => void): void;
}

// Consolidamos el contexto aquí también
export type ComponentContext = any; 

// Definimos los tipos de creación basados en la interfaz 'Component'
export type ComponentConstructor = new (ctx: ComponentContext) => Component;
export type ComponentFactory = (ctx: ComponentContext) => Component;

// El Creador unificado
export type ComponentCreator = ComponentConstructor | ComponentFactory;
2. Limpia el fichero del Router
Ahora que component.model.ts tiene la lógica pesada, el fichero del router solo tiene que importar y definir lo que es específico de la navegación (el Provider y la Route).

3. ¿Son compatibles?
Sí, lo son, pero con un matiz importante:

BaseComponent vs Component: Si tus clases de componente extienden de una clase BaseComponent, asegúrate de que esa clase implemente la interfaz Component.

Ejemplo: class MyComponent extends BaseComponent implements Component { ... }

Flexibilidad: La definición ComponentConstructor = new (ctx: any) => Component es más segura que la anterior porque garantiza que lo que salga del new siempre tendrá un método .render(), que es lo que el loader y la hidratación necesitan obligatoriamente.

Beneficios de esta limpieza:
Adiós al shadowing: Ya no tendrás errores de "ComponentCreator ya está definido".

Mantenimiento: Si mañana decides que todos los componentes deben tener un método onResize(), solo lo añades en component.model.ts y automáticamente el Router, el Loader y la Hidratación lo reconocerán.

Tipado fuerte: TypeScript dejará de quejarse en el loader.resolve porque el ComponentCreator que devuelve el import() dinámico coincidirá exactamente con lo que espera el instantiate.

Un último consejo: En component.model.ts, asegúrate de que BaseComponent (si lo sigues usando) y la interfaz Component no se pisen. Lo ideal es que BaseComponent sea la implementación base y Component la descripción del contrato.
